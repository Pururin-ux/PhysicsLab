# PhysicsLab Roadmap

Этот roadmap готовит репозиторий к дальнейшей разработке. Он не является задачей на миграцию и не требует менять текущий сайт.

Источник: `PhysicsLab_research_report.pdf`, версия от 15 мая 2026.

## 1. Current Baseline

Текущий проект - статический прототип на HTML/CSS/JS:

- `pages/index.html`;
- `pages/mechanics-menu.html`;
- `pages/mechanics.html`;
- `css/style.css`;
- `js/mechanics.js`;
- `assets/`.

На этапе подготовки нельзя менять текущий сайт без отдельной задачи. Документы в `docs/` задают правила для будущего развития.

## 2. Product Direction

PhysicsLab развивается как интерактивный учебник-тренажер по физике для школьников Беларуси.

Ключевая формула:

живой стиль + строгая методика + проверяемая физика + официальный белорусский контекст + удобная компонентная архитектура.

## 3. No-Migration Rule for Preparation Stage

На подготовительном этапе:

- не переносить проект в Astro, React, Next, Vue или другой стек;
- не создавать `src/` без явной задачи;
- не переписывать текущие страницы;
- не менять `pages/`, `css/`, `js/`, `assets/`;
- не менять визуальный стиль текущего сайта.

Разрешено:

- добавлять документацию;
- описывать будущую архитектуру;
- составлять чеклисты;
- готовить правила контента и ассетов;
- инвентаризировать риски без изменения сайта.

## 4. Recommended Future Stack

Отчет рекомендует для будущего масштабирования:

- Astro;
- MDX;
- TypeScript;
- Tailwind;
- интерактивные islands-компоненты.

Причина: PhysicsLab является content-first проектом с большим числом глав, формул, задач и точечных интерактивных симуляций.

Эта рекомендация не означает, что миграцию нужно делать сейчас.

## 5. Phase 0: Repository Preparation

Цель: сделать проект понятным для будущих агентов и разработчиков.

Deliverables:

- `docs/design-and-learning-rules.md`;
- `docs/content-rules.md`;
- `docs/agent-instructions.md`;
- `docs/physics-review-checklist.md`;
- `docs/asset-policy.md`;
- `docs/roadmap.md`.

Acceptance criteria:

- правила практичны;
- явно запрещено копирование чужого контента;
- новый физический контент требует review status;
- текущий сайт не изменен.

## 6. Phase 1: Content Inventory

Цель: понять, что уже есть и чего не хватает.

Задачи:

- описать существующие страницы;
- перечислить существующие главы;
- найти битые ссылки на будущие главы;
- составить карту тем механики;
- отметить места, где есть физический контент без review status;
- отдельно отметить ассеты без лицензии или происхождения.

Выход:

- `docs/content-inventory.md` в будущей задаче;
- список приоритетных глав;
- список physics-review рисков;
- список asset-review рисков.

## 7. Phase 2: Learning Model and Content Schema

Цель: подготовить структуру будущих глав, задач и квизов.

Будущие решения:

- схема frontmatter главы;
- схема задачи;
- схема квиза;
- статусы проверки;
- список misconceptions;
- список skills;
- правила exam A/B;
- правила retrieval prompts.

Важно: физический контент должен быть оригинальным и иметь `authorReviewRequired` или `needs-physics-review`.

## 8. Phase 3: Component Architecture Planning

Цель: спроектировать компоненты до миграции, но не внедрять их без отдельной задачи.

Будущие компоненты:

- `FormulaCard`;
- `VariableTable`;
- `WorkedExample`;
- `MisconceptionCard`;
- `MiniQuiz`;
- `ExamTask`;
- `CharacterBubble`;
- `SimulationCard`;
- `RetrievalCard`.

Правило: если паттерн повторяется больше двух раз, после миграции или в рамках явной архитектурной задачи его нужно вынести в компонент.

## 9. Phase 4: First Course Priorities

Приоритеты из отчета:

1. Механика.
2. Электродинамика.
3. МКТ и термодинамика.
4. Оптика и элементы СТО.
5. Квантовая и ядерная физика.

Механика должна закрыть:

- физический язык: СИ, графики, векторы, проекции;
- путь, перемещение, скорость;
- равноускоренное движение;
- графики x(t), v(t), a(t);
- силы и законы Ньютона;
- вес, трение, реакция опоры;
- работа, мощность, энергия;
- импульс и законы сохранения.

## 10. Phase 5: First Simulation Priorities

Первая очередь симуляций:

- равномерное движение;
- равноускоренное движение;
- графики движения;
- сложение векторов;
- второй закон Ньютона;
- энергия;
- газовые законы;
- электрическая цепь;
- линза;
- фотоэффект.

Каждая симуляция должна следовать циклу:

Predict -> Manipulate -> Observe -> Explain -> Transfer.

## 11. Phase 6: Review and QA Infrastructure

Будущие проверки:

- проверка битых ссылок;
- accessibility smoke tests;
- проверка контраста;
- тесты физических функций;
- валидация схем задач и глав;
- проверка наличия review status;
- проверка asset register;
- проверка, что нет нелицензионных ассетов.

## 12. Phase 7: Migration Decision

Миграцию обсуждать только после:

- инвентаризации контента;
- утверждения схем глав и задач;
- понимания списка компонентов;
- решения по ассетам;
- согласия владельца проекта.

Возможная целевая структура из отчета:

  docs/
  public/assets/
    characters/
    memes/
    diagrams/

Это ориентир, не текущая задача.

## 13. Definition of Ready for New Physics Content

Новая глава или задача готова к разработке, если есть:

- тема;
- targeted skills;
- класс или уровень;
- prerequisites;
- exam relevance;
- список формул;
- список misconceptions;
- план worked/faded examples;
- статус проверки;
- источники или указание `source: original`.

## 14. Definition of Done for New Physics Content

Новый физический контент готов к публикации только если:

- он оригинальный;
- формулы проверены;
- единицы проверены;
- ответы проверены;
- графики подписаны;
- feedback объясняет ошибки;
- есть review status;
- ассеты имеют лицензионную запись;
- экран не перегружает внимание;
- мобильная версия читается;
- accessibility базово соблюдена.
