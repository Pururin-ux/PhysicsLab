# PhysicsLab Mascot Production Plan

Дата: 2026-05-15

Статус: план подготовки production-ready web-ассетов маскота. Этот документ не разрешает использовать текущие concept PNG напрямую в интерфейсе.

## 1. Почему concept PNG нельзя вставлять напрямую

В `assets/mascot/concepts/` лежат пять concept-изображений:

- `guide-calm.png`;
- `guide-encouraging.png`;
- `guide-surprised.png`;
- `guide-thinking.png`;
- `guide-warning.png`.

Они полезны как visual reference для маскота PhysicsLab, но не готовы для production UI.

Причины:

- файлы имеют формат PNG, но режим `RGB`;
- alpha-канала нет;
- реальной прозрачности нет;
- шахматная подложка является частью изображения;
- кроп и масштаб не утверждены как UI-система;
- вес файлов слишком большой для регулярного использования в интерфейсе;
- нет финальной проверки на сходство с существующей IP;
- нет набора адаптированных размеров для `CharacterBubble`, onboarding и маленьких реакций.

Правило: concept PNG нельзя подключать в `/lab-preview/`, production-страницы или компоненты как финальные web-ассеты.

## 2. Целевая структура

Финальные web-версии должны жить отдельно от concept-файлов:

```text
assets/mascot/
  concepts/
    guide-calm.png
    guide-encouraging.png
    guide-surprised.png
    guide-thinking.png
    guide-warning.png
  ui/

      full.webp
      bubble.webp
      avatar.webp
    encouraging/

      full.webp
      bubble.webp
      avatar.webp
    surprised/
      full.png
      bubble.png

      avatar.webp
    thinking/
      full.png
      bubble.png
```

На текущем этапе в `ui/` создаются только папки и `.gitkeep`. Реальные финальные изображения не создавать из concept PNG, пока нет качественного ручного или утверждённого pipeline для удаления фона.

## 3. Состояния маскота

Целевые состояния соответствуют `CharacterBubble` prop `state`:

| State | Роль | Где использовать |
| --- | --- | --- |
| `calm` | спокойная подсказка | вступление, мягкое объяснение, onboarding |
| `encouraging` | поддержка | feedback после ошибки, следующий шаг, итог |
| `surprised` | удивление | неожиданный факт, разворот ожидания |
| `thinking` | размышление | сложный шаг, выбор закона, связь формулы и графика |
| `warning` | предупреждение | ловушка, misconception, экзаменационный риск |

Состояние должно помогать учебному смыслу. Нельзя менять state только ради эмоции, если это не поддерживает текущий блок.

## 4. Целевые размеры

Для каждого состояния нужны три базовые версии.

### `full`

Ориентир:

- примерно 800-1000 px по высоте;
- PNG с настоящим alpha-каналом;
- optional WebP после утверждения PNG.

Использование:

- большие подсказки;
- onboarding;
- secondary hero;
- отдельные character moments.

Ограничение: `full` не должен становиться главным элементом учебного экрана рядом с формулой, графиком или задачей.

### `bubble`

Ориентир:

- примерно 320-480 px по высоте;
- PNG с настоящим alpha-каналом;
- optional WebP после утверждения PNG.

Использование:

- `CharacterBubble`;
- `MisconceptionCard`;
- объяснение сложного шага;
- спокойная подсказка в главе.

Это основной формат для интерфейса.

### `avatar`

Ориентир:

- 96-160 px;
- PNG с настоящим alpha-каналом;
- optional WebP после утверждения PNG.

Использование:

- маленькие реакции;
- статусы;
- quiz feedback;
- компактные подсказки.

Avatar не должен быть единственным носителем смысла. Рядом всегда нужен текст.

### Optional `webp`

WebP-версии нужны после утверждения PNG:

- для оптимизации загрузки;
- для responsive UI;
- как производные файлы от финального PNG, а не от текущего concept PNG.

PNG остаётся source-of-truth для прозрачности и проверки качества.

## 5. Требования к финальным ассетам

Финальные файлы должны соответствовать требованиям:

- настоящий alpha-канал;
- чистый читаемый силуэт;
- без текста;
- без логотипов;
- без интерфейсных рамок;
- без шахматной подложки;
- одинаковый кроп между состояниями;
- одинаковая логика масштаба головы, плеч и рук;
- персонаж не занимает больше внимания, чем формула, график или задача;
- разумный вес файла после оптимизации;
- no hover-only meaning: эмоция не должна быть доступна только при наведении;
- mobile layout не ломается из-за ширины или высоты изображения.

Ориентиры по весу после оптимизации:

- `avatar`: до 40-80 KB;
- `bubble`: до 120-250 KB;
- `full`: до 250-500 KB.

Это ориентиры, а не абсолютные лимиты. Качество силуэта и читаемость важнее механического сжатия.

## 6. Alt-текст и доступность

Если изображение смысловое, нужен alt-текст:

```text
Гид PhysicsLab предупреждает о типичной ошибке в чтении графика.
```

Если изображение декоративное, оно должно быть скрыто от screen reader:

```html
aria-hidden="true"
```

Правило: важная информация не должна передаваться только выражением лица, цветом или позой маскота. Реплика рядом должна объяснять смысл обычным текстом.

## 7. Mobile requirements

На mobile маскот должен:

- не выталкивать формулу за пределы экрана;
- не создавать горизонтальный скролл;
- не перекрывать график или controls;
- занимать компактную колонку или небольшой avatar-слот;
- оставлять текст реплики главным содержанием;
- иметь fallback, если изображение не загрузилось.

Для `CharacterBubble` основной mobile-формат - `bubble` или `avatar`, но не `full`.

## 8. Workflow подготовки

Безопасный workflow:

1. Утвердить character bible и роль маскота.
2. Выбрать concept-файл как визуальный reference.
3. Подготовить финальный cutout вручную или через утверждённый pipeline.
4. Проверить настоящий alpha-канал.
5. Выровнять кроп между состояниями.
6. Экспортировать `full`, `bubble`, `avatar`.
7. Оптимизировать PNG без потери силуэта.
8. Создать WebP как производные версии.
9. Обновить `assets/mascot/LICENSES.md`.
10. Проверить `CharacterBubble` на desktop, tablet и mobile.
11. Запустить `npm run check`, `npm run build`, `npm run screenshots`.

## 9. Что нельзя делать

Нельзя:

- автоматически вырезать шахматную подложку без отдельной задачи и проверки качества;
- подключать concept PNG напрямую в `CharacterBubble`;
- использовать concept PNG в hero или учебных блоках как финальный asset;
- добавлять похожего на существующего аниме-персонажа маскота;
- вставлять маскота рядом с каждой репликой;
- делать персонажа ярче и важнее формулы, графика или задачи;
- добавлять новые web-версии без записи в license register.

## 10. Критерий готовности

Production-ready mascot asset считается готовым, если:

- есть настоящая прозрачность;
- нет шахматной подложки;
- кроп согласован между состояниями;
- файл оптимизирован;
- есть license record;
- есть понятное состояние;
- есть alt-политика;
- проверен mobile layout;
- маскот помогает учебному UX, а не становится декором.
