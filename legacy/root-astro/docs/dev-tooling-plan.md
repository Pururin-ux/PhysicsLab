# PhysicsLab Dev Tooling Plan

Дата: 2026-05-15

Цель: описать безопасный набор dev-инструментов и интеграций для PhysicsLab после перехода на Astro + MDX + Tailwind + TypeScript.

Ограничение: этот документ не требует менять текущий код. Инструменты и npm scripts добавлять только на этапе миграции, когда в проекте появятся `package.json`, Astro и `src/`.

## 1. Главный принцип

Инструменты должны защищать три вещи:

- текущий темный желтый аниме/мемный стиль PhysicsLab;
- работоспособность интерактивов, особенно p5.js-визуализации;
- физическую корректность нового контента.

Инструмент не считается полезным, если он ускоряет работу ценой визуального шума, нелицензионных ассетов или непроверенной физики.

## 2. Инструменты для разработки сайта

### 2.1 Playwright Screenshots

Назначение:

- фиксировать baseline текущих страниц перед миграцией;
- сравнивать Astro-версию с HTML/CSS/JS-референсом;
- ловить пустой canvas, сломанную сетку, переполнение формул, исчезнувшие ассеты.

Что проверять:

- `pages/index.html` или будущий `/`;
- оглавление механики;
- глава с p5.js-визуализацией;
- состояния квиза;
- состояния progress dashboard;
- mobile и desktop.

Минимальные viewport:

- `360x740`;
- `390x844`;
- `768x1024`;
- `1280x900`;
- `1440x1000`.

Правило:

Screenshots нужны до переключения маршрутов. Новая версия не должна визуально уходить в стерильный корпоративный EdTech.

### 2.2 Mobile Checks

Назначение:

- защитить мобильную версию от регрессий;
- проверить крупную формулу, таблицы, quiz options, scenario buttons и canvas.

Проверять:

- нет горизонтального скролла страницы;
- формулы переносятся или имеют controlled overflow;
- touch targets достаточно крупные;
- нет hover-only важных подсказок;
- canvas имеет адекватную высоту;
- персонаж и реплика не перекрывают учебный текст;
- quiz feedback помещается на экране.

Рекомендуемый инструмент:

- Playwright projects с мобильными viewport;
- ручной smoke pass в браузере перед крупным merge.

### 2.3 Accessibility Checks

Назначение:

- удержать минимум WCAG AA;
- не потерять keyboard/focus/reduced-motion при миграции.

Рекомендуемые инструменты после миграции:

- `@axe-core/playwright` для автоматических a11y smoke tests;
- Playwright keyboard navigation smoke tests;
- ручная проверка `prefers-reduced-motion`;
- contrast checks для токенов и критичных блоков.

Проверять:

- видимый focus state;
- интерактивы доступны с клавиатуры или имеют fallback;
- смысл не передается только цветом;
- alt-текст у смысловых изображений;
- декоративные изображения скрыты от screen reader;
- нет мелкого низкоконтрастного текста;
- motion можно снизить.

### 2.4 Image Optimization

Назначение:

- оптимизировать утвержденные локальные ассеты;
- не тянуть картинки из интернета;
- сохранять качество персонажей, схем и hero-изображений.

Допустимые инструменты после миграции:

- Astro image pipeline;
- `sharp` для локальной оптимизации;
- `svgo` для собственных SVG-схем;
- ручной экспорт WebP/AVIF из утвержденных исходников.

Правила:

- оптимизировать только локальные, утвержденные ассеты;
- не скачивать изображения из интернета автоматически;
- не заменять ассет без записи в `assets/LICENSES.md`;
- для персонажей сохранять исходник и экспортные версии;
- для учебных схем сохранять читаемость подписей.

### 2.5 Lint and Format

Назначение:

- держать Astro/MDX/TypeScript код предсказуемым;
- уменьшить случайный churn;
- не ловить стиль руками в каждом PR.

Рекомендуемые инструменты после миграции:

- `astro check`;
- TypeScript strict checks;
- ESLint для TypeScript/JS;
- Prettier;
- `prettier-plugin-astro`;
- `prettier-plugin-tailwindcss`, если Tailwind classes начнут расползаться;
- Markdown lint только в мягком режиме, чтобы не ломать учебный текст.

Правила:

- formatter не должен массово переформатировать старые HTML-файлы без отдельной задачи;
- линтер не должен требовать переписать p5.js sketch одновременно с переносом;
- стиль учебного текста важнее механического "идеального" markdown.

### 2.6 Tests for Physics Functions

Назначение:

- проверять вычислительную физику после миграции;
- отделить физические формулы от UI;
- снизить риск ошибок в задачах, симуляциях и квизах.

Рекомендуемый инструмент:

- `vitest`.

Где тестировать:

- `src/lib/physics/kinematics.ts`;
- будущие модули для динамики, энергии, электричества, термодинамики;
- форматирование численных ответов;
- округление;
- unit conversion helpers.

Минимальные тесты для кинематики:

- равномерное движение: `x = x0 + vt`;
- скорость по графику: `v = dx / dt`;
- площадь под `v(t)` как перемещение;
- знаки скорости и перемещения;
- нулевые и предельные случаи.

Важно:

Unit tests не заменяют physics review. Новый физический контент все равно получает `authorReviewRequired: true` или `reviewStatus: "needs-physics-review"`.

## 3. Допустимые MCP и интеграции

### 3.1 Browser / Playwright

Допустимо:

- открывать локальный dev server;
- делать screenshots;
- проверять responsive layout;
- проверять canvas-pixel smoke;
- кликать квизы, кнопки сценариев, progress controls;
- запускать accessibility smoke checks, если инструмент доступен.

Ограничения:

- использовать прежде всего для локального проекта;
- не использовать для скачивания картинок;
- не обходить asset policy через браузер.

### 3.2 GitHub

Допустимо:

- смотреть PR, issues, CI и review comments;
- публиковать ветки и PR по явному запросу;
- читать логи GitHub Actions;
- проверять failing checks.

Ограничения:

- не пушить без явной задачи;
- не закрывать review comments без фактического исправления;
- не добавлять секреты и токены в репозиторий.

### 3.3 Figma

Допустимо только при наличии макетов или прямого запроса владельца проекта.

Можно:

- читать предоставленный Figma-макет;
- сверять layout с макетом;
- переносить дизайн-токены, если они утверждены;
- использовать Figma как reference, а не как источник случайной переработки стиля.

Нельзя:

- подключать Figma "просто чтобы красиво переделать";
- заменять текущую идентичность PhysicsLab стерильной дизайн-системой;
- создавать макеты, которые конфликтуют с `docs/design-and-learning-rules.md`.

## 4. Запрещенные MCP и интеграции

### 4.1 Скачивание картинок из интернета

Запрещено:

- искать и скачивать картинки через MCP;
- брать кадры аниме;
- брать чужие мемы;
- брать изображения из Google Images;
- автоматически добавлять найденные картинки в `assets/`.

Причина:

PhysicsLab запрещает нелицензионные картинки, кадры аниме, чужих персонажей и мемы без лицензии.

### 4.2 Неизвестные MCP с широкими правами

Запрещено подключать или использовать неизвестные MCP, которые:

- имеют широкий доступ к файловой системе;
- могут скачивать и добавлять файлы;
- могут менять репозиторий без прозрачного diff;
- имеют сетевой доступ без понятной цели;
- обходят review workflow.

Если инструмент не нужен для конкретной задачи и не понятен по правам, его не использовать.

### 4.3 Инструменты, добавляющие ассеты без записи в asset policy

Запрещены любые инструменты или workflow, которые:

- добавляют изображения без `assets/LICENSES.md`;
- генерируют персонажей без утверждения;
- заменяют существующие ассеты без записи источника;
- создают мемы без проверки прав;
- встраивают base64-картинки в код без учета лицензии.

Ассет без происхождения считается risk item, а не готовым ресурсом.

## 5. Как делать аниме-маскотов

### 5.1 Character Bible

Перед добавлением маскота нужен отдельный character bible.

Рекомендуемый будущий файл:

```text
docs/character-bible.md
```

Минимальное содержание:

- имя персонажа;
- роль в обучении: наставник, скептик, лаборант, экзаменатор, мемный комментатор;
- характер и тон;
- что персонаж может говорить;
- что персонаж не должен говорить;
- визуальные признаки;
- палитра;
- разрешенные эмоции и позы;
- запрет на сходство с существующими аниме/IP;
- список утвержденных файлов;
- правила частоты появления на странице.

### 5.2 Ручное утверждение ассетов

Маскот не попадает в проект автоматически.

Workflow:

1. Создать или получить кандидатный ассет вне production-кода.
2. Проверить, что он не похож на существующего персонажа.
3. Проверить, что он не является кадром, перерисовкой или derivative work без прав.
4. Утвердить вручную владельцем проекта.
5. Добавить запись в `assets/LICENSES.md`.
6. Только после этого интегрировать файл в UI.

### 5.3 assets/LICENSES.md

Каждый файл маскота должен иметь запись:

```markdown
## characters/mika-neutral.png

- Author: PhysicsLab / project author
- Source: original
- License: Project-owned
- Date added: YYYY-MM-DD
- Approved by: project owner
- Notes: Original character, not based on existing anime IP.
```

Для сгенерированного файла добавить:

```markdown
- Generation tool: tool name
- Prompt policy: no named artist, no existing character, no anime frame
- Review: manually approved
```

### 5.4 Роль Codex

Codex не должен сам добывать или придумывать production-ассеты маскотов без отдельного явного запроса.

Codex может:

- интегрировать уже утвержденные файлы;
- обновлять alt-текст;
- подключать персонажа через `CharacterBubble`;
- проверять, есть ли запись в `assets/LICENSES.md`;
- предложить структуру character bible.

Codex не должен:

- скачивать персонажей из интернета;
- использовать кадры аниме;
- генерировать похожих на известных персонажей маскотов;
- добавлять файл в `assets/` без утверждения и license record.

## 6. NPM Scripts после перехода на Astro

Эти scripts добавить только после создания Astro-проекта.

Рекомендуемый набор:

```json
{
  "scripts": {
    "dev": "astro dev",
    "build": "astro build",
    "check": "astro check && tsc --noEmit",
    "lint": "eslint . && prettier --check .",
    "test": "vitest run",
    "test:e2e": "playwright test",
    "test:a11y": "playwright test --project=a11y",
    "screenshots": "playwright test --project=screenshots"
  }
}
```

### 6.1 dev

Запускает локальный Astro dev server.

Требование:

- не заменять текущий статический сайт до этапа переключения маршрутов.

### 6.2 build

Собирает production build.

Требование:

- build должен падать, если content schema не проходит проверку.

### 6.3 check

Запускает Astro и TypeScript checks.

Требование:

- контентные схемы должны ловить отсутствие review status для физического контента.

### 6.4 lint

Проверяет код и форматирование.

Требование:

- не использовать `--write` в CI;
- не форматировать старые reference HTML массово без отдельной задачи.

### 6.5 test

Запускает unit tests.

Основной фокус:

- физические функции;
- unit conversion;
- rounding;
- validation schemas.

### 6.6 test:e2e

Запускает Playwright e2e.

Основной фокус:

- навигация;
- quiz interactions;
- progress localStorage;
- p5.js сценарии;
- absence of blank canvas.

### 6.7 test:a11y

Запускает accessibility checks.

Основной фокус:

- axe smoke;
- focus states;
- keyboard path;
- reduced motion;
- semantic alt behavior.

### 6.8 screenshots

Создает или сравнивает screenshots.

Основной фокус:

- visual parity со старым HTML/CSS/JS;
- mobile regressions;
- формулы, таблицы, квизы, p5 canvas.

## 7. CI после GitHub-подключения

Минимальный GitHub Actions pipeline после миграции:

1. install dependencies;
2. `npm run check`;
3. `npm run lint`;
4. `npm run test`;
5. `npm run build`;
6. `npm run test:e2e` для ключевых страниц;
7. `npm run test:a11y` для smoke coverage.

Screenshots в CI включать осторожно: сначала как artifact, потом как blocking check только после стабилизации baseline.

## 8. Чего не делать сейчас

Пока проект не мигрировал на Astro:

- не добавлять `package.json` только ради scripts;
- не устанавливать Playwright;
- не переписывать HTML;
- не переносить p5.js;
- не менять `pages/`, `css/`, `js/`, `assets/`;
- не добавлять новые ассеты;
- не использовать MCP для поиска изображений.

Текущая задача - только документация.

## 9. Короткий вывод

Безопасный tooling для PhysicsLab должен быть консервативным: Playwright для визуального и мобильного контроля, axe для accessibility smoke, ESLint/Prettier/Astro check для кода, Vitest для физической логики после миграции, GitHub для CI и PR, Figma только при наличии макетов.

Самое важное ограничение: никакой инструмент не должен добавлять ассеты без ручного утверждения и записи в `assets/LICENSES.md`. Маскоты и мемная эстетика остаются частью PhysicsLab, но production-файлы проходят character bible, asset approval и license record.
