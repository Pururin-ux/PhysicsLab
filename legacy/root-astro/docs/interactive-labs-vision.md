# PhysicsLab Interactive Labs Vision

Дата: 2026-05-15

Статус: vision document для будущей разработки PhysicsLab 2.0. Это не спецификация готовой реализации и не миграция текущего сайта.

## 1. Главный принцип

PhysicsLab должен развиваться как интерактивный учебник-лаборатория, а не как обычная страница с текстом и картинками.

Главный принцип интерактивной лаборатории:

```text
Ученик меняет параметр -> одновременно меняются формула, график, анимация,
численные значения, задача и объяснение.
```

Изменение параметра должно сразу показывать физический смысл: что именно поменялось, почему это видно на графике, как это связано с формулой и как такая же идея появляется в задаче ЦЭ/ЦТ.

Интерактивность не нужна ради эффекта. Она нужна, чтобы ученик увидел связь между:

- физической ситуацией;
- формулой;
- единицами измерения;
- графиком;
- численным ответом;
- типичной ошибкой;
- экзаменационным вопросом.

## 2. Что такое Interactive Lab

Interactive Lab - это учебный блок, где один физический закон исследуется через управляемую модель.

В минимальном варианте лаборатория включает:

- **Ползунки** для 1-3 главных параметров на первом уровне.
- **Живые формулы**, которые пересчитываются при изменении параметров.
- **Подсветку изменяемых членов формулы**, чтобы ученик видел, какой символ сейчас отвечает за изменение.
- **Графики в реальном времени** с осями, единицами и понятным смыслом наклона, площади или точки.
- **Анимацию физической ситуации**, которая показывает не декоративную сцену, а физическое состояние системы.
- **Проверку единиц**, чтобы ученик связывал число с размерностью.
- **Предсказание перед действием**, чтобы интерактив не превращался в случайное дергание слайдеров.
- **Объяснение после действия**, чтобы наблюдение превращалось в вывод.
- **Перенос на задачу**, чтобы лаборатория готовила к решению, а не оставалась игрушкой.

Лаборатория должна отвечать на вопрос ученика: "Что произойдет, если я изменю этот параметр, и почему?"

## 3. Педагогический цикл

Каждая эталонная лаборатория строится вокруг цикла:

```text
Предскажи -> измени -> наблюдай -> объясни -> реши задачу
```

### Предскажи

Перед изменением параметра ученик делает короткое предположение.

Примеры:

- "Если скорость увеличить в 2 раза, что станет с координатой через то же время?"
- "Как изменится наклон графика x(t)?"
- "Что будет с графиком v(t), если скорость постоянна?"

### Измени

Ученик меняет один параметр через ползунок, переключатель или числовой ввод.

Важно: на первом уровне не давать слишком много ручек сразу. Если ученик видит пять слайдеров без маршрута, лаборатория начинает повышать cognitive load.

### Наблюдай

Интерфейс одновременно обновляет:

- численное значение;
- формулу;
- график;
- анимацию;
- короткую подпись к наблюдению.

Наблюдение должно быть конкретным: "наклон графика стал больше", "точка прошла больший путь", "координата изменилась на vt".

### Объясни

После наблюдения ученик получает вопрос или короткий feedback.

Примеры:

- "Почему график x(t) стал круче?"
- "Какой член формулы отвечает за изменение?"
- "Где на графике видно значение скорости?"

### Реши задачу

Финальный шаг переносит идею в учебную или exam-style задачу. Задача должна быть оригинальной, простой для первого уровня и иметь статус `authorReviewRequired` или `needs-physics-review`.

## 4. Технический принцип

Физика должна быть отделена от UI.

Будущая структура:

```text
src/lib/physics/*
src/components/simulations/*
```

`src/lib/physics/*` хранит вычислительную физику:

- чистые TypeScript-функции;
- формулы;
- unit conversion;
- проверку диапазонов;
- округление;
- подготовку точек для графиков;
- тестируемые helpers.

`src/components/simulations/*` хранит UI интерактивных лабораторий:

- ползунки и controls;
- SVG/Canvas-графики;
- анимационные панели;
- связь с `FormulaCard`, `QuizCard`, `SimulationPanel`;
- accessible labels;
- reduced-motion fallback.

Правило: если формулу можно протестировать без браузера, она должна жить в `src/lib/physics/*`, а не внутри компонента.

После миграции физические функции должны покрываться unit tests. Но тесты не заменяют physics review: новый учебный контент все равно требует проверки автора.

## 5. Первая эталонная лаборатория: UniformMotionLab

Первая reference-лаборатория для PhysicsLab 2.0:

```text
UniformMotionLab
```

Тема: равномерное прямолинейное движение.

Статус demo-контента при создании: `authorReviewRequired`.

Текущая Astro foundation использует `UniformMotionLab` как первый эталонный шаблон интерактивной лаборатории:

```text
src/lib/physics/kinematics.ts
src/components/simulations/UniformMotionLab.astro
```

Принцип реализации: физика считается в чистых TypeScript-функциях, а компонент только показывает связь между параметрами, формулой, графиками, движением точки и учебным вопросом. Будущие лаборатории должны наследовать этот подход: интерактивность служит пониманию, а не украшению.

В текущем демо формула, график `x(t)`, график `v(t)`, положение точки, численные значения и мини-проверка обновляются вместе. Это задаёт стандарт для будущих лабораторий: изменение параметра должно сразу показывать физический смысл в нескольких представлениях.

### Параметры

Первый уровень:

- `x0` - начальная координата;
- `v` - скорость;
- `t` - время.

Дополнительные параметры можно раскрывать позже, но они не должны мешать первому сценарию.

### Формула

Главная формула:

```text
x = x0 + vt
```

Требования:

- подсвечивать `x0`, `v`, `t`, когда ученик меняет соответствующий параметр;
- показывать SI units;
- объяснять, что `vt` - изменение координаты за время `t`;
- явно отделять координату `x` от пути, если это важно для задачи.

### Графики

Лаборатория должна показывать:

- график `x(t)`;
- график `v(t)`.

Требования:

- оси подписаны;
- единицы видимы;
- наклон графика `x(t)` связан со скоростью;
- график `v(t)` показывает постоянную скорость;
- на mobile графики не ломают ширину и не требуют горизонтального скролла страницы.

### Анимация

Анимация показывает движение точки вдоль оси.

Требования:

- точка не должна быть декоративной;
- положение точки связано с текущим `x`;
- скорость визуально читается через изменение положения за время;
- при `prefers-reduced-motion` должна быть статическая или пошаговая версия.

### Проверка единиц

Лаборатория должна помогать видеть размерности:

```text
x: м
x0: м
v: м/с
t: с
vt: м
```

Если ученик вводит или видит несовместимые единицы, интерфейс должен объяснить проблему спокойно, без экзаменационного давления.

### Mini-check

Мини-проверка после исследования:

- один вопрос на предсказание;
- один вопрос на связь формулы и графика;
- один вопрос на типичную ошибку.

Feedback должен быть формирующим: объяснить ход мысли, а не просто сказать "верно" или "неверно".

### Exam-style task

Задача в стиле подготовки к ЦЭ/ЦТ должна:

- быть оригинальной;
- не копировать задания РИКЗ, учебники или сборники;
- проверять тот же принцип, который ученик исследовал в лаборатории;
- иметь понятные единицы и округление;
- быть помечена `authorReviewRequired` или `needs-physics-review`.

## 6. Будущие лаборатории

После `UniformMotionLab` можно развивать линейку лабораторий:

- равноускоренное движение;
- графики движения;
- второй закон Ньютона;
- сила трения;
- энергия;
- импульс;
- газовые законы;
- закон Ома;
- цепи;
- линзы;
- фотоэффект.

Приоритет стоит давать темам, где интерактивность действительно снижает непонимание:

- связь графика и формулы;
- знаки величин;
- единицы измерения;
- предельные случаи;
- типичные ловушки ЦЭ/ЦТ;
- переход от наблюдения к задаче.

## 7. Ограничения

Интерактивность не должна становиться визуальным шумом.

Обязательные ограничения:

- На первом уровне лаборатории использовать не больше 1-3 главных параметров.
- Сложные параметры раскрывать постепенно через advanced mode, details или отдельный шаг.
- Mobile должен быть полноценным, а не урезанным просмотром desktop-версии.
- Reduced-motion fallback обязателен для анимаций и движущихся графиков.
- Физические формулы, задачи, объяснения и ответы требуют проверки автора.
- Нельзя скрывать физический смысл за эффектами, неоном, персонажем или анимацией.
- Нельзя делать слайдер без вопроса, график без осей, анимацию без наблюдаемой величины.
- Нельзя добавлять реальные ассеты, мемы или персонажей без asset policy и license record.
- CharacterBubble может помогать с ловушкой или поддержкой, но не должен конкурировать с формулой, графиком и задачей.

Если лаборатория выглядит "круто", но ученик не может объяснить, что именно изменилось и почему, дизайн нужно упрощать.

## 8. Future Stack Options

Эти варианты не являются обязательством внедрить все сразу. Они описывают безопасное направление для будущей разработки.

### Astro для страниц и MDX

Astro подходит для:

- учебных страниц;
- маршрутов глав;
- статического контента;
- MDX-объяснений;
- SEO и быстрой загрузки.

MDX можно использовать для глав, когда появится контентная схема с review status и правилами физического контента.

### Svelte или React islands для лабораторий

Интерактивные лаборатории можно делать как islands:

- Svelte island, если нужен компактный state и реактивность для симуляций;
- React island, если проект выберет React ecosystem для компонентов и тестов.

Выбор нужно делать после первого prototype spike для `UniformMotionLab`, а не абстрактно.

### TypeScript functions для физики

Физические вычисления должны быть TypeScript-функциями:

- независимыми от DOM;
- покрытыми тестами;
- понятными для physics review;
- пригодными для переиспользования в графике, задаче и feedback.

### SVG/Canvas для графиков

Возможные варианты:

- SVG для простых графиков, подписей, осей и доступной структуры;
- Canvas для более тяжелых анимаций и частого перерисовывания;
- гибрид SVG + Canvas, если нужно совместить crisp labels и плавную визуализацию.

Для первых лабораторий предпочтительно начинать проще: SVG или легкий Canvas без тяжелых библиотек, если этого достаточно.

### KaTeX позже для сложных формул

KaTeX можно добавить позже, когда появятся:

- реальные главы;
- сложные формулы;
- правила отображения формул на mobile;
- проверка overflow;
- единый стиль `FormulaCard`.

На раннем этапе нельзя добавлять KaTeX только ради внешнего вида, если формулы пока простые и читаемые.

## 9. Definition of Done для интерактивной лаборатории

Лаборатория считается готовой к review, если:

- есть один главный учебный фокус;
- параметры, формула, график, анимация и задача связаны между собой;
- есть цикл `Предскажи -> измени -> наблюдай -> объясни -> реши задачу`;
- графики имеют оси, единицы и подписи;
- формулы имеют переменные, единицы и область применимости;
- есть misconception или trap card;
- есть formative feedback;
- есть mobile layout без горизонтального скролла;
- есть reduced-motion fallback;
- вычисления отделены от UI;
- физический контент помечен `authorReviewRequired` или `needs-physics-review`;
- нет нелицензионных ассетов и случайных картинок из интернета.

## 10. Короткий вывод

PhysicsLab 2.0 должен ощущаться как Dark Anime Lab, но работать как серьезная учебная лаборатория: ученик меняет параметр и сразу видит физику в формуле, графике, движении, числах, объяснении и задаче.

Главная мера качества - не количество эффектов, а момент понимания: ученик может сказать, что изменилось, где это видно и как применить это в задаче ЦЭ/ЦТ.
